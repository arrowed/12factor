Whats it all about?

  • Maximising isolation and portability
  • Enable scaling

~310 docker containers, ~12 concurrent development teams, 14 development stacks, staging, demo, and production environments, with 4,500 releases

1: Codebases

  • 1 applicaton = 1 codebase
  • 1 codebase is deployed to many environments
  • Promote versions through the environments
  • Extract common functions to libraries
We have put a lot of work into establishing base layers of our apps

  • Be explicit
  • Dont rely on operating system packages
  • One method of installing your dependencies

3: Configuration

Environment settings/variables stored outside of source control
  • Usernames, passwords
  • Connection strings/resource identifiers
  • Instance specific identifiers
  • Independent of each other

Could you open source your code without fear of leaked secrets?

Things that dont vary from environment to environment can be handled in code- di config/other constants

The 12factor authors like environment variables, whatever works for you. Could be pulled/pushed in at runtime or deploytime, from a database or other config management system Self discovery and self documenting configuration goes a long way. Who knew that the url had to end in a / anyway?

Limit the _kinds_ of environments, it'll simplify

4: Backing services are resources

  • All backing services treated equally

Extension of the configuration idea, but promoting the resource-as-a-parameter

Can you change your database server without a code change

5: Separate your pipeline build, release and deploy

  • Build it
  • Release it
    1. Combine your config with your artifact
    2. Improve your version number to a release identifier
  • Deploy it
Build
  1. Get your dependencies
  2. Compile it
  3. Run your tests
  4. Identify a version number
  5. Produce an artifact

Your release number may be a container identifier, config hash/version, deploy timestamp, environment key+version

Your release identifier needs to be Immutable

Deployment should be simple. See coming slides

6: Keep it stateless

  1. All your data lives in backing services
  2. If you need it, dont assume its existence for any subsequent process.
  3. May limit scalability options
  4. Sticky sessions

Sticky sessions hold session data in ram/local storage and rely on

subsequent requests to be processed on the same instance.

7: Port binding

  • Expose your service on a port
  • Be self contained and run anywhere
  • Become the backing service for something else
  • Utilise routing and registration/service discovery
Service registration occurs during deployment

8: Concurrency

  • Drawn from the unix process model
  • Can still have your own internal threadpools
  • Allows for horizontal scaling inside a server resource
  • Take advantage of existing mechanisms for fault control and resilience

May have some haters on this, but this is where the effort starts to pay off

Dont have to run instance per app, can share load to

This is something IOOF takes advantage of. Heavy services are spread evenly over extra capacity nodes

Dont build your own reliability controls. Harder to scale, get right and housekeep

9: Ebrace disposability

  • Start quickly
  • Stop cleanly and gracefully
  • Fail fast
  • Be robust
    1. use job queuing (be reentrant)
    2. use retrys (be idempotent)

Fast to boot means quick to respond to scaling requests, config changes

Longer between starts means other services may need longer timeouts/retry intervals

Stop taking requests when shutting down, whist finishing current committed workload

Being disposable means you can move your app in a production setting without consequence (ie start new instance, update registration, and decommission old. you can because you arent relying on anything)

what if the os shuts down mid process?

10: Dev/Prod parity

  1. Continually deploy
    • Keep development and prod as close as possible.
    • More frequently the releases, the smaller and easier they become.
  2. Backing service similarity
    • Dont rely on a feature that isnt in production
    • At least keep parity in testing environments
  • Release frequently. Stop living in the past.
  • More release iteratons == resilience
  • Dont have to learn 2 things
  • Hit by a truck
  • Parity can be in violation of 2: dependencies, ie hard to setup a local mongo cluster
  • Will always be some gap, ie dont run on production hardware in development

    • 11: Logs are streams

    • Limit your use of local os resources (files)
    • Let something eat your logs/event stream
    • Limit your housekeeping
    What if you have two instances of an app running to spread load, on the same server resource?
    enables centralisation of log streams for * fast diagnosis * trend analysis / graphing/dashboarding later on * non naive app alerting

    12: Management tasks are one off tasks

    • Running db migrations on app startup is slow
    • Baking in one off things into apps may mean dead/stale code persists
    Can get race conditions when starting instances
    • IOOF dont do great at this, but we are getting better.
    • Complex migrations (applying fee upgrades) will be done as a separate app allowing for development iteration and promotion through our release process
    • Only a handful of people have backing data store access (relational/document databases) and traceability is important.
